# 私有组件库从零搭建完整指南

基于 `deer-plus` 项目架构的完整搭建步骤

## 项目概述

本指南将指导你从零开始搭建一个现代化的私有组件库，采用与 `deer-plus` 项目相同的技术栈和架构设计。

## 技术栈深度解析

### 为什么选择这些技术？

**Vue 3.5+**：
- **优势**：组合式 API 提供更好的逻辑复用，更好的 TypeScript 支持，更小的包体积
- **适用场景**：企业级应用、需要良好类型支持的项目
- **替代方案**：React、Svelte、SolidJS

**TypeScript**：
- **优势**：类型安全、更好的 IDE 支持、减少运行时错误
- **适用场景**：大型项目、团队协作、需要长期维护的项目
- **学习成本**：中等，但收益显著

**Vite 7.0+**：
- **优势**：极快的冷启动、热更新、开箱即用的配置
- **核心原理**：基于原生 ES 模块，无需打包即可开发
- **对比 Webpack**：构建速度提升 10-100 倍

**pnpm**：
- **优势**：磁盘空间节省、安装速度快、严格的依赖管理
- **核心特性**：内容可寻址存储、符号链接管理依赖
- **对比 npm/yarn**：更高效的依赖管理和更少的磁盘占用

## 技术栈选择

### 核心技术
- **Vue 3.5+** - 现代化前端框架
- **TypeScript** - 类型安全的开发体验
- **Vite 7.0+** - 快速构建工具
- **pnpm** - 高效的包管理器

### 组件库架构
- **Monorepo 结构** - 多包项目管理
- **ECharts 集成** - 数据可视化组件
- **Element Plus 扩展** - UI组件基础
- **VitePress 文档** - 组件文档系统

### 开发工具
- **Vitest** - 单元测试框架
- **ESLint + Prettier** - 代码规范
- **Husky** - Git hooks管理

## 第一步：项目初始化和基础配置

### 1.1 创建项目目录结构
```bash
# 创建项目根目录
mkdir my-component-library
cd my-component-library

# 创建基础目录结构
mkdir -p packages/{components,theme-chalk,utils,locale}
mkdir -p docs/{components,examples,guide,public/images}
```

**目录结构设计原理**：
- **packages/**: Monorepo 的核心，每个子包独立管理
- **components/**: 组件实现，按功能模块划分
- **theme-chalk/**: 主题系统，支持样式定制
- **utils/**: 工具函数，避免代码重复
- **locale/**: 国际化支持
- **docs/**: 文档系统，使用 VitePress

**为什么选择这种结构？**
- **模块化**：每个功能独立，便于维护和测试
- **可扩展性**：易于添加新功能模块
- **团队协作**：不同团队可以负责不同模块

### 1.3 配置 pnpm workspace
创建 `pnpm-workspace.yaml`：

```yaml
packages:
  - 'packages/*'
  - 'docs'
```

**pnpm workspace 原理**：
- **符号链接**：所有包共享 node_modules，通过符号链接连接
- **依赖提升**：共同的依赖被提升到根目录的 node_modules
- **版本一致性**：确保所有包使用相同版本的依赖

**优势对比**：
- **vs npm/yarn workspace**：pnpm 的磁盘空间使用更少，安装更快
- **独立安装**：每个包可以独立安装，但共享依赖
- **开发体验**：修改一个包，其他包立即生效

### 3.2 创建安装器
创建 `packages/my-library/make-installer.ts`：

```typescript
import type { App, Plugin } from 'vue'
import type { ComponentSize } from '@element-plus/components'

export type SFCWithInstall<T> = T & Plugin

export const withInstall = <T>(comp: T) => {
  (comp as SFCWithInstall<T>).install = (app: App) => {
    const { name } = comp as any
    app.component(name, comp)
  }
  return comp as SFCWithInstall<T>
}

export const makeInstaller = (components: Plugin[] = []) => {
  const install = (app: App) => {
    components.forEach((c) => app.use(c))
  }

  return {
    install,
  }
}
```

**安装器架构深度解析**：

**withInstall 函数原理**：
```typescript
export const withInstall = <T>(comp: T) => {
  (comp as SFCWithInstall<T>).install = (app: App) => {
    const { name } = comp as any
    app.component(name, comp)
  }
  return comp as SFCWithInstall<T>
}
```

**作用**：
- 为组件添加 `install` 方法
- 支持 `app.use(Component)` 的全局注册方式
- 保持组件的类型安全

**makeInstaller 函数**：
```typescript
export const makeInstaller = (components: Plugin[] = []) => {
  const install = (app: App) => {
    components.forEach((c) => app.use(c))
  }
  return { install }
}
```

**批量安装机制**：
- 接收组件数组，统一注册
- 支持按需导入和全量导入
- 提供灵活的组件管理方式

**使用场景对比**：

**全量导入**：


**按需导入**：

**优势**：
- **Tree Shaking 友好**：未使用的组件不会打包进最终代码
- **类型安全**：完整的 TypeScript 支持

**构建输出格式**：

### 5.2 示例：Button 组件
创建 `packages/components/button/src/button.vue`：

```vue
<template>
  <button 
    class="my-button"
    :class="[
      `my-button--${type}`,
      `my-button--${size}`,
      {
        'is-disabled': disabled,
        'is-loading': loading
      }
    ]"
    :disabled="disabled || loading"
    @click="handleClick"
  >
    <span v-if="loading" class="my-button__loading">
      <i class="my-icon-loading"></i>
    </span>
    <span class="my-button__content">
      <slot></slot>
    </span>
  </button>
</template>

<script setup lang="ts">
import { withDefaults } from 'vue'

defineOptions({
  name: 'MyButton'
})

interface Props {
  type?: 'primary' | 'success' | 'warning' | 'danger' | 'info'
  size?: 'large' | 'default' | 'small'
  disabled?: boolean
  loading?: boolean
}

const props = withDefaults(defineProps<Props>(), {
  type: 'primary',
  size: 'default',
  disabled: false,
  loading: false
})

const emit = defineEmits<{
  click: [event: MouseEvent]
}>()

const handleClick = (event: MouseEvent) => {
  if (!props.disabled && !props.loading) {
    emit('click', event)
  }
}
</script>
```

## 第七步：配置测试环境

### 7.1 安装测试依赖

**测试工具链深度解析**：

**Vitest**：
- **优势**：与 Vite 生态完美集成，极快的测试运行速度
- **特性**：支持 ESM、TypeScript、热重载
- **对比 Jest**：更快的启动速度，更好的开发体验

**@vue/test-utils**：
- **作用**：Vue 组件的测试工具库
- **核心 API**：`mount`、`shallowMount`、`wrapper`
- **功能**：模拟用户交互、测试组件生命周期

**happy-dom**：
- **作用**：轻量级的 DOM 实现
- **优势**：比 JSDOM 更快，内存占用更小
- **适用场景**：组件单元测试

### 7.2 Vitest 配置
创建 `vitest.config.ts`：

```typescript
import { defineConfig } from 'vitest/config'
import vue from '@vitejs/plugin-vue'

export default defineConfig({
  plugins: [vue()],
  test: {
    environment: 'happy-dom',
    globals: true,
  },
})
```

**Vitest 配置详解**：

**globals: true**：
- **作用**：自动注入全局测试函数（describe、it、expect 等）
- **优势**：无需手动导入，简化测试代码

**environment: 'happy-dom'**：
- **作用**：指定测试运行环境
- **选择理由**：轻量级，适合组件测试
- **替代方案**：jsdom（功能更全但更重）

**插件配置**：
```typescript
plugins: [vue()]
```
- **作用**：支持 .vue 单文件组件的测试
- **原理**：使用与开发环境相同的构建配置

### 7.3 示例测试文件
创建 `packages/components/button/__tests__/button.test.ts`：

```typescript
import { mount } from '@vue/test-utils'
import { describe, expect, it } from 'vitest'
import Button from '../src/button.vue'

describe('Button.vue', () => {
  it('renders slot content', () => {
    const wrapper = mount(Button, {
      slots: {
        default: 'Click me'
      }
    })
    expect(wrapper.text()).toBe('Click me')
  })

  it('emits click event', async () => {
    const wrapper = mount(Button)
    await wrapper.trigger('click')
    expect(wrapper.emitted()).toHaveProperty('click')
  })

  it('disabled button does not emit click', async () => {
    const wrapper = mount(Button, {
      props: {
        disabled: true
      }
    })
    await wrapper.trigger('click')
    expect(wrapper.emitted()).not.toHaveProperty('click')
  })
})
```

**测试代码深度解析**：

**测试用例设计原则**：

**渲染测试**：
```typescript
it('renders slot content', () => {
  const wrapper = mount(Button, {
    slots: { default: 'Click me' }
  })
  expect(wrapper.text()).toBe('Click me')
})
```
- **目的**：验证组件是否正确渲染内容
- **方法**：使用 `slots` 选项传递插槽内容
- **断言**：检查渲染后的文本内容

**事件测试**：
```typescript
it('emits click event', async () => {
  const wrapper = mount(Button)
  await wrapper.trigger('click')
  expect(wrapper.emitted()).toHaveProperty('click')
})
```
- **目的**：验证组件是否正确触发事件
- **方法**：使用 `trigger` 模拟用户交互
- **异步处理**：使用 `async/await` 等待事件处理完成

**属性测试**：
```typescript
it('disabled button does not emit click', async () => {
  const wrapper = mount(Button, {
    props: { disabled: true }
  })
  await wrapper.trigger('click')
  expect(wrapper.emitted()).not.toHaveProperty('click')
})
```
- **目的**：验证属性对组件行为的影响
- **方法**：通过 `props` 选项设置组件属性
- **反向断言**：验证禁用状态下不触发事件

**测试最佳实践**：
- **单一职责**：每个测试用例只验证一个功能
- **描述性命名**：测试名称清晰描述测试目的
- **隔离性**：测试之间相互独立，不依赖执行顺序
